LATAM Autoship Stack Architecture Not all LATAM payment methods support recurring. Local cards are the most reliable recurring method across all markets and offer the highest approval rates. Pix Automático and SmartPix enable Brazil recurring (Aug 2025). OXXO, Boleto, PagoEfectivo, and other cash/voucher methods are one-time only — acquire new subscribers via these methods but route them to card or Pix for ongoing autoship. This two-track architecture (acquisition via APMs + recurring via card/Pix) is the standard LATAM subscription model.
Direct Selling Expertise dLocal explicitly mentions direct selling as a supported use case in their Amway partnership press release. The stated challenges — lower card penetration, high cross-border processing costs, fragmented APM rails — are exactly what Unicity faces. dLocal's local acquiring model in LATAM is the primary lever: domestic card transactions processed as local (not cross-border) improve approval rates and eliminate the 3–5% FX premium that cross-border adds to every charge.

1
Customer Selects Pix at Checkout
dLocal checkout presents SmartPix enrollment option. Customer provides Pix key (CPF, phone, email, or random key). One-time Pix authorization establishes the token. No QR code scanning on subsequent charges.
POST /payments {payment_method_id: "XA", country: "BR", enrollment: {...}}
2
One-Time Authorization — Token Issued
Customer authorizes via their banking app (Bradesco, Itaú, Nubank, etc.). dLocal returns enrollment_id — a token identifying the approved subscription. This token enables all future MIT charges without customer action.
GET /enrollments/{id} → {status: "AUTHORIZED", enrollment_id}
3
Monthly Autoship — MIT via enrollment_id
Unicity submits monthly charge using enrollment_id. No QR code, no customer re-authorization. Variable amounts supported (SmartPix advantage over Pix Automático which requires fixed amounts). Pix Automático: submit 2–10 days in advance per BCB rules.
POST /payments {enrollment_id, amount: 285, currency: "BRL"}
4
Decline — Retry Available
On failure (e.g. insufficient funds), dLocal's retry service can re-attempt using the same enrollment_id. Align retry timing with Unicity's +0/+3/+5/+7 cadence. Webhook fires on all state changes.
POST /payments/{id}/retry → webhook: payment.completed | failed

Account Updater on Card Decline
On soft decline from expired/replaced card, dLocal Account Updater fetches updated card credentials from the networks. Retry with updated card. Unicity dunning cadence applies for hard declines and non-recoverable failures.
payment_failed → account_updater.refresh(token) → retry

Token Portability — Key Unknown dLocal holds card tokens in their vault as the merchant-of-record in each LATAM market. dLocal's IP and data rights on tokens are not publicly documented. Before any significant subscriber base is enrolled on dLocal card tokens, the token portability SLA must be confirmed: (1) export format, (2) export timeline on termination, (3) whether tokens are stored as network tokens (Visa Token Service / Mastercard MDES) or as PSP-proprietary tokens. PSP-proprietary tokens are non-portable and would require subscriber re-enrollment on migration.

Recommended Approach

Start dLocal in markets where PayU has the weakest coverage or highest cross-border rates. Brazil (SmartPix) and Colombia (Amway reference case) are the best entry points. Run dLocal in parallel with PayU initially — don't migrate all existing PayU subscribers until token portability is resolved on both sides. dLocal's single-contract model significantly reduces operational overhead once fully onboarded.

Migration Risk

Any existing PayU LATAM card tokens must be assessed for portability before migration planning. If PayU tokens are non-portable (PSP-proprietary format), existing autoship subscribers would need to re-enter card details during migration. Quantify this subscriber re-enrollment risk before committing a dLocal migration timeline. Start fresh subscriber enrollment on dLocal — migrate existing only after token portability is resolved on both ends.
